


# Тестовое задание: Backend + Nginx (reverse proxy)

Простое веб-приложение на Python, вынесенное за Nginx в качестве
reverse proxy. Оба сервиса запускаются через `docker-compose` в
единой сети с драйвером `bridge`.

## Запуск проекта
Копируем репозиторий:
```bash
git clone https://github.com/Mr-p1gnum/TZ-for-EM
```
или
```bash
git clone https://gitlab.mrp1gnum.ru/mr-p1gnum/tz-for-em
```
Затем
```bash
# из корня проекта
docker-compose up -d --build
```

Собрать заново без кэша (например, после правок в `app.py`):

```bash
docker-compose build --no-cache backend
docker-compose up -d
```

Остановить и удалить контейнеры:

```bash
docker-compose down
```

## Проверка работоспособности

```bash
curl http://localhost
```

Ожидаемый ответ (без `HEADER` в `.env`):

```
Hello from Effective Mobile!
```

Если в `.env` задана переменная `HEADER`, ответ будет содержать
дополнительную строку:

```
Hello from Effective Mobile!
Deployed via Docker Compose
```

## Архитектура

```
                 ┌────────────────────┐
  curl / браузер │                    │
 ───────────────▶│   nginx (порт 80)  │
                  │  reverse-proxy    │
                  └─────────┬─────────┘
                            │ proxy_pass
                            │ http://backend:8080
                            ▼
                  ┌────────────────────┐
                  │  backend (Python)  │
                  │  порт 8080         │
                  │  (не публикуется   │
                  │   наружу)          │
                  └────────────────────┘
```

Наружу публикуется **только порт 80** контейнера с nginx. Backend
доступен исключительно внутри docker-сети `proxy-back` по имени
сервиса `backend`.

## Backend

Минимальный HTTP-сервер на чистом Python (`http.server`, без
сторонних фреймворков и зависимостей).

**`backend/app.py`**

- слушает `0.0.0.0:8080`;
- на `GET /` возвращает текст `"Hello from Effective Mobile!"`
  с кодом `200`;
- если задана переменная окружения `HEADER`, добавляет её
  значение отдельной строкой в ответ;
- на любой другой путь возвращает `404 Not Found`.

**`backend/Dockerfile`**

- базовый образ — `python:3.12-alpine` (лёгкий образ);
- код приложения копируется в `/app`;
- создаётся отдельный непривилегированный пользователь
  `backenduser` (группа `backendgroup`), от имени которого и
  запускается процесс — контейнер не работает от `root`;
- контейнер стартует командой `python app.py`;
- порт `8080` объявлен через `EXPOSE`, но наружу через
  `docker-compose` не публикуется — доступ к нему возможен только
  из docker-сети `proxy-back`.

## Nginx

Официальный образ `nginx:alpine`. Конфигурация подключается через
volume — конфиги на хосте монтируются внутрь контейнера в режиме
"только чтение" (`:ro`), сам образ при этом не пересобирается.

**`nginx/nginx.conf`** — основной конфиг

- задаёт пользователя, число worker-процессов, `pid`-файл;
- отключает вывод версии nginx в заголовках ответа
  (`server_tokens off`);
- настраивает логи (`access_log`, `error_log`);
- подключает все конфиги виртуальных хостов из директории
  `conf.d/*.conf` — это позволяет держать основной конфиг чистым
  и добавлять/убирать сервисы отдельными файлами, не трогая
  `nginx.conf`.

**`nginx/conf.d/backend.conf`** — конфиг для backend-сервиса

- слушает `80` порт;
- единственный `location /` проксирует все запросы на
  `http://backend:8080` (имя сервиса `backend` резолвится
  через встроенный Docker DNS внутри сети `proxy-back`);
- используется `proxy_http_version 1.1` с очисткой заголовка
  `Connection`, чтобы между nginx и backend работал keepalive;
- прокидываются заголовки `Host`, `X-Real-IP`,
  `X-Forwarded-For` — backend видит реальный IP клиента и
  исходный хост запроса;
- заданы таймауты подключения/чтения (`proxy_connect_timeout`,
  `proxy_read_timeout`) и отключена буферизация ответа
  (`proxy_buffering off`).

## Docker Compose

Корневой `docker-compose.yml` поднимает два сервиса в одной сети
`proxy-back` с драйвером `bridge`:

| Сервис          | Контейнер       | Образ / сборка              | Порты                    | Сеть          |
|-----------------|-----------------|------------------------------|---------------------------|---------------|
| `revers-proxy`  | `revers-proxy`  | `nginx:alpine`               | `80:80` (наружу)          | `proxy-back`  |
| `backend`       | `backend`       | сборка из `./backend`        | не публикуется наружу     | `proxy-back`  |

Особенности конфигурации:

- **`revers-proxy`** монтирует локальную папку `./nginx` в
  `/etc/nginx` контейнера (`:ro`) — это и есть подключение
  основного конфига вместе с директорией `conf.d`;
- **`backend`** собирается из своего `Dockerfile` (`build.context:
  ./backend`), значение переменной `HEADER` подтягивается из файла
  `.env` через `env_file`;
- у обоих сервисов заданы понятные имена контейнеров
  (`container_name`) и политика перезапуска
  `restart: unless-stopped`;
- сеть `proxy-back` объявлена внутри самого compose-файла с
  явным драйвером `bridge` — сервисы видят друг друга по имени,
  внешний доступ есть только к тому, что явно проброшено через
  `ports`.

## Переменные окружения

Backend поддерживает необязательную переменную:

| Переменная | Обязательна | По умолчанию | Описание                                   |
|------------|-------------|--------------|---------------------------------------------|
| `HEADER`   | нет         | `""`         | Дополнительная строка, добавляемая к ответу |

Задаётся в файле `.env` в корне проекта (сам файл в `.gitignore`,
в репозиторий не попадает):

```env
HEADER=Deployed via Docker Compose
```
